🚀 Team Git Workflow – 4 Developers, 1 Project

A professional Git branching strategy for clean, collaborative development.
👥 Designed for teams of 4 developers working on a single project.

🌳 Branching Strategy

main        → 🚀 Production-ready only
dev         → 🧪 Shared development
feature/*   → 👨‍💻 Individual feature branches

🗂️ Branch Roles

🔹 main

  • ✅ Always deployable
  • 🔒 Protected
  • 🔁 Only merged via Pull Requests (PRs) from dev

🔹 dev

  • 🧪 Main development branch
  • 🧵 All features get merged here before main

🔹 feature/your-name-task

  • 👨‍🔧 One per developer per feature
  • 🌱 Created from dev
  • 🔀 Merged into dev via PR

🔁 Daily Workflow

🆕 Step 1: Create your feature branch

git checkout dev
git pull origin dev
git checkout -b feature/your-name-task

💻 Step 2: Do your work, then commit and push

git add .
git commit -m "Added [feature name]"
git push -u origin feature/your-name-task

🔄 Step 3: Keep your branch updated with dev

git checkout dev
git pull origin dev

git checkout feature/your-name-task
git merge dev

✅ Step 4: Open a Pull Request to merge into dev

🚀 Merging to main

  • All features are merged into dev
  • Code is tested and reviewed
  • CI tests pass

Then open a PR from dev → main on GitHub

🔐 Branch Protection Rules

BranchProtection Settings
main✅ Require PRs, reviews, CI tests
dev⚠️ PRs preferred, soft rules
feature/*💻 Developer-controlled

🙌 Final Notes

  • Stay in your feature branch
  • PR into dev, not main
  • Keep main clean and deployable

"Good branching keeps bugs in check and devs sane."
